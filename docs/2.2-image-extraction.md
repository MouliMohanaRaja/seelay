# Design note — Step 2.2: image extraction (T3 OCR → T4 vision)

Pre-implementation. Grounds the 2.2 build in the pipeline that already exists
(`lib/resolve/`), so no new machinery is invented and the state machine is untouched.
The one-line shape: **image → OCR text → the existing text parser → TMDB → scorer**, and
only if that comes up empty, **image → vision LLM → TMDB → scorer**.

## 1. Where OCR runs

Server-side, inside the async resolution — never in the capture request. Capture already
returns instantly (Law 2); resolution runs in `after()` via `scheduleResolution`, and the
item is in `resolving` while it does. OCR is **tier T3** in `lib/resolve/`, reached only
for image captures after T0–T2 produce nothing (they always will for images — no URL, no
text). The pipeline loads the stored bytes for the capture (via the storage helper +
`payload_image_ref`) and hands them to **tesseract.js (WASM)**, which runs in-process — no
new service, per the single-stack architecture decision.

*Constraint to watch:* tesseract cold-start (WASM core + `eng` traineddata) plus
recognition must fit the serverless function's execution budget. If it's too slow or
memory-heavy, that pressure pushes work onto T4 — which is exactly what R1's tripwire is
for. Flag at first real run, don't pre-optimize.

## 2. What text OCR returns

A noisy, multi-line blob of everything legible in the screenshot — the title if we're
lucky, but also app chrome ("Netflix", "Add to My List"), usernames, timestamps,
captions. T3's raw output is the recognized **text plus tesseract's per-line/word
confidence** (0–100). It is not a clean title and must not be treated as one.

## 3. How it feeds T0–T2

T3 does not invent a new matching path — it produces text that **re-enters the waterfall
at T2** (`extractT2`), reusing the parsing that already works: strip service words, drop
filler, isolate a title query. Because the blob is multi-line, T3 selects a small set of
**candidate lines** (rank by OCR confidence + prominence/font size; skip obvious chrome),
turns each into a query through T2, and keeps the best-scoring TMDB match across them.

Two deliberate details:
- If a candidate line contains a recognizable URL, it may re-enter at **T0/T1** instead
  (deterministic, high-confidence) — a nice-to-have, not the primary path.
- T2's "— from X" who-hint parser should be **suppressed or distrusted** on OCR text: a
  screenshot that happens to contain the word "from" must not fabricate provenance
  (Law 4). Provenance for screenshots stays user-supplied.

## 4. When OCR is considered successful

Two gates, not one. "OCR ran" ≠ "we identified it."
- **OCR-level:** non-empty text above a mean-confidence floor. Empty or garbage → OCR
  failed outright.
- **Resolution-level:** an OCR-derived candidate, run through TMDB + the existing scorer,
  clears at least the confirm bar (`CONFIRM_AT = 0.45`, i.e. `needs_confirm` or better).

OCR is "successful" only when both hold. An OCR pass that yields text but no match at or
above the confirm bar counts as **not sufficient** — same as the existing waterfall rule
that a tier "succeeds" only if it produces a confident candidate.

## 5. When to invoke the vision fallback (T4)

Only when T3 is not sufficient (empty/garbage OCR, or best OCR-derived match below the
confirm bar) — identical gating to the rest of the waterfall (a tier runs only if the
previous found nothing confident). T4 is the **Intelligent Fallback**: the vendor-neutral,
env-configured LLM (this is its first use in the project), called with a **vision-capable
model** on the image bytes, returning structured `{title, year?, media_type?}` candidates
that then go through TMDB + the scorer like any other candidate. **Every T4 call is logged
and counted** — it is the LLM-free-rate metric (A11) and the budget line.

R1's budget corollary applies verbatim: if OCR alone misses the 7/10 bar but OCR+T4
clears it, that is a **pass** — revise the A11 expectation, never weaken the accuracy bar.

## 6. How confidence propagates into the state machine

No new states, no state-machine change. Images flow `raw → resolving → (retrying if TMDB
is flaky) → resolved | needs_confirm | needs_hint`, exactly as text does. Confidence
propagates through the **existing** `candidate.confidence`, which the scorer multiplies by
TMDB `matchQuality`:

- **T3 (OCR):** `candidate.confidence` is derived from tesseract's mean confidence,
  normalized to 0–1 and **capped modestly** (well below the deterministic tiers — T0 ≈
  0.98, T2 = 0.6 — because OCR is inherently noisy). Consequence, and it's the honest one:
  an OCR'd title that is an **exact** TMDB match hits the scorer's `matchQuality === 1`
  floor (0.8) → `resolved`; a fuzzy match stays under `RESOLVE_AT` → `needs_confirm`
  (ask, don't guess — Law 3).
- **T4 (vision):** `candidate.confidence` = the model's self-rating, similarly capped;
  `llmUsed = true` and `tierUsed = "T4"` are recorded (feeding A11 and `metadata.tier`).

OCR/vision failure does **not** use the `retrying` state — that's reserved for transient
*upstream* (TMDB) errors. A weak OCR result escalates *down the waterfall* to T4; a weak
T4 result lands honestly in `needs_hint`, where the one-word hint loop (already built)
rescues it.

## Non-goals (explicitly out of scope for 2.2)

Named so the implementation doesn't drift into them. Each is a plausible accuracy lever
we are deliberately *not* pulling yet; most become fair game only if the 2.2 verify
underperforms and R1 fires.

- **No image preprocessing** — no cropping, deskew/rotation correction, contrast/enhance,
  or upscaling before OCR. Feed the stored bytes as-is. (Already the PLAN 2.2 fence.)
- **No multilingual OCR tuning** — `eng` traineddata only; no per-language model loading,
  script detection, or transliteration. Regional-title coverage stays TMDB's job (R6),
  not OCR's. A non-Latin screenshot that OCR can't read escalates to T4, as designed.
- **No OCR-text persistence** — the recognized blob is a transient input to matching, not
  stored on the capture or the item. `captures` stays the immutable image; only the
  resolved entity + `tierUsed` land on the item. (Re-OCR on a hint retry is fine.)
- **No duplicate / near-duplicate detection** — the same screenshot uploaded twice makes
  two captures and two items. Dedupe is a separate concern, not this step.
- **No layout/region intelligence** — no object detection, poster-vs-chrome
  segmentation, or bounding-box heuristics beyond tesseract's own line confidence used
  for candidate-line ranking. No training or fine-tuning of anything.
- **No T4 model shopping** — one env-configured vision model (PLAN 2.2 fence); no
  provider comparison or per-image model selection.
- **No new capture surface or gallery** — 2.1's single upload control is the only entry
  point; no multi-select, no camera-roll scan (that stays R4's deferred tripwire).

## What does not change

The state machine, the scorer's thresholds, the resolver interface, the hint loop, and
the receipt. 2.2 adds two tiers inside `lib/resolve/` and an image-loading branch in the
pipeline — nothing else. Success is measured by the PLAN 2.2 verify: 10 real screenshots,
≥ 7/10 correct top candidate, with per-tier stats (OCR-alone vs. needed-T4) recorded.
